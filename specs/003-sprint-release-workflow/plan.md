# Implementation Plan: Sprint–Release Workflow Orchestrator

**Branch**: `feature/003-sprint-release-workflow` | **Date**: 2026-06-17
**Spec**: `specs/003-sprint-release-workflow/spec.md`

## Summary

NodeToolbox will gain a new polling scheduler (`sprintReleaseScheduler`) that
orchestrates the development issue lifecycle across four configurable Jira projects.
The scheduler watches `customfield_10201` (sub-status) on dev issues: when the value
changes to "Ready for System Integration Test" the dev issue is transitioned to Done
and a QE handoff comment is posted; when it changes to "Ready for UAT" a BT handoff
comment is posted. Config-only issues (labelled `no-testing-required`) close at the
sub-status trigger with no handoff. A parallel watch detects fixVersion date changes
and syncs the dev team's sprint end date via the Jira Agile API. Defect intake is
triggered when QE/BT apply a `defect-intake` label on one of their issues; NodeToolbox
creates a linked dev issue (original stays Done, assignee inherited). The DoR gate
checks dev sprint issues daily for missing QE/BT criteria fields. All configuration
lives in a new `sprintRelease.teamProfiles` block in `toolbox-proxy.json`. The data
model carries a `teamProfileId` on every record to support multi-team expansion later.

## Technical Context

**Language/Version**: Node.js 18+ / JavaScript (no TypeScript)
**Primary Dependencies**: Express (existing), native `https` module via `httpClient.js`
**Storage**: `toolbox-proxy.json` in AppData (new `sprintRelease` top-level key) +
  in-memory Maps for runtime state (same pattern as all existing schedulers)
**Testing**: Jest — unit tests with mocked `httpClient.js`; integration tests against
  live Jira sandbox
**Target Platform**: Windows desktop (NodeToolbox server process, port 5555)
**Project Type**: Node.js server with Express API routes + background schedulers
**Performance Goals**: Poll cycle completes in < 10 seconds per team profile; handoff
  fires within one poll interval (≤ 5 min) of sub-status change; sprint date syncs
  within one business day of fixVersion date change
**Constraints**: No new npm dependencies; all Jira calls through existing
  `makeJiraApiRequest`; no inbound webhook endpoint needed (polling chosen)
**Scale/Scope**: Single team profile (UI); data model multi-team-ready; dozens of
  issues per sprint; one fixVersion per sprint cycle

### Phase C — Sub-status polling + Done transition

**Goal**: Implement the core poll loop — detect `customfield_10201` changes,
transition dev issue to Done, post handoff comment.

Business logic in `sprintReleaseOrchestrator.js`:
- `detectSubStatusChanges(issues, lastHandoffMap, profileConfig)` — pure; returns events
- `executeDevIssueDone(issueKey, jiraConfig, profileConfig)` — fetch transitions, fire Done
- `postHandoffComment(issueKey, handoffType, featureSummary, jiraConfig)` — POST comment
- `buildHandoffComment(issueKey, handoffType, featureKey, featureSummary)` — pure; returns comment body

Tests (unit):
- `detectSubStatusChanges` with changelog containing QE sub-status → returns QE event
- `detectSubStatusChanges` with `no-testing-required` label → returns bypass event (no handoff)
- `detectSubStatusChanges` already-seen issue → returns nothing (idempotent)
- `buildHandoffComment` returns correct structure for QE and BT types
- `executeDevIssueDone` called with mocked transitions → calls correct transition ID

Integration test:
- Full Scenario 1 and 2 from quickstart.md against live Jira sandbox

### Phase D — FixVersion date sync + business day math

- `updateSprintEndDate(sprintId, newEndDate, jiraConfig)` — calls Agile API
- `findSprintByName(sprintName, boardId, jiraConfig)` — queries Agile API

## Key design decisions (summary)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Event detection | Polling (5-min interval) | Matches all existing schedulers; no inbound route needed |
| Sub-status trigger | `customfield_10201` changelog watch | Same pattern as fixVersion change detection in scopeChangeScheduler |
| Sprint update API | Jira Agile API `/rest/agile/1.0/sprint/{id}` | Existing auth; `makeJiraApiRequest` handles it |
| Storage | AppData JSON + in-memory Maps | No database in project; same pattern as all schedulers |
| Defect model | New issue (original stays Done) | Preserves delivery credit; enables separate defect metrics |
| Defect trigger | `defect-intake` label on QE/BT issue | QE/BT stay in their own Jira tooling |
| Multi-team | Single-team UI; `teamProfileId` on all records | Safe expansion path without over-building today |
| Business day math | Pure JS helper (~20 lines) | No date library warranted for one function |
| Handoff delivery | Jira comment + optional outbound webhook | Reuses `httpClient.js`; no new delivery infrastructure |
